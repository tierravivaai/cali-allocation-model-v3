**Findings**

1. The most serious issue is the hidden sticky `equality_mode`. The app initializes it at [src/app.py:79](/Users/pauloldham/Documents/cali-allocation-model-unscale-v3-peer-review/src/app.py#L79), the sliders update only weights at [src/app.py:217](/Users/pauloldham/Documents/cali-allocation-model-unscale-v3-peer-review/src/app.py#L217), and the calculator ignores those weights when equality mode is on at [src/cali_model/calculator.py:128](/Users/pauloldham/Documents/cali-allocation-model-unscale-v3-peer-review/src/cali_model/calculator.py#L128). In practice, the main controls can look active while the model is still producing a pure equal split.

2. The floor control and the calculator disagree on who is eligible when high-income countries are excluded. The UI uses “exclude all high income” logic at [src/app.py:296](/Users/pauloldham/Documents/cali-allocation-model-unscale-v3-peer-review/src/app.py#L296), while the calculator preserves high-income SIDS at [src/cali_model/calculator.py:119](/Users/pauloldham/Documents/cali-allocation-model-unscale-v3-peer-review/src/cali_model/calculator.py#L119). If the user picks a floor in that gap, the allocator falls back to equality at [src/cali_model/calculator.py:53](/Users/pauloldham/Documents/cali-allocation-model-unscale-v3-peer-review/src/cali_model/calculator.py#L53), which is a material outcome change.

3. The party table misstates units for share columns. The labels say percent at [src/app.py:940](/Users/pauloldham/Documents/cali-allocation-model-unscale-v3-peer-review/src/app.py#L940), but the formatting shows 0-1 fractions. That is easy to miss in development and easy to misread in policy review.

4. The public calculator defaults in [src/cali_model/calculator.py:101](/Users/pauloldham/Documents/cali-allocation-model-unscale-v3-peer-review/src/cali_model/calculator.py#L101) do not match the repository’s documented default behavior in [README.md:29](/Users/pauloldham/Documents/cali-allocation-model-unscale-v3-peer-review/README.md#L29) and the surrounding README text. That makes off-app usage brittle.

**What Works Well**

The repo has strong foundations. The separation between data loading, allocation logic, sensitivity analysis, and reporting is good, and the test suite is unusually thoughtful for a policy-support tool: it checks math invariants, party alignment, UI behavior, baseline logic, and special cases like EU and SIDS handling. That gives the model much more credibility than a typical dashboard-only implementation.

The project also does a good job of making its normative assumptions visible. The README, tests, and UI text all try to explain why the model behaves the way it does, which is exactly what a peer-reviewable allocation tool should do.
